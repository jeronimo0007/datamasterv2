✅ Etapa 1 · Extração de dados

Da fonte ao barramento

  • Scripts Python (multi-fonte): conectores e jobs para APIs do core, arquivos de parceiros, exportações históricas e scripts/generate_data.py como simulador de carga. Adaptação de schema: src/data_ingestion/transaction_adapters.py normaliza JSON de simulador, fila (Event Hubs / Kinesis) ou CSV para o dicionário do modelo.
  • Event Hub Producer (streaming): publicação assíncrona com metadados (event_id, timestamp) e particionamento — par AWS em Kinesis Data Streams ou MSK. Repo: src/data_ingestion/event_hub_producer.py.
  • Azure Data Factory (Terraform) (Glue Workflow / Step Functions na AWS): pipelines (lookup → copy → trigger), secrets no Key Vault (Secrets Manager / SSM), destino ADLS Gen2 (S3). Infra declarativa junto de Event Hubs / Kinesis.

  • Abra src/data_ingestion/event_hub_producer.py e aponte send_batch / metadados (narrativa Event Hubs / Kinesis).

✅ Etapa 2 · Ingestão de dados

Streaming, batch e Lambda

  • Streaming — Event Hubs (Kinesis / MSK) + consumer: consumer group, checkpoint em Blob Storage (S3 + metadados / client library Kinesis); backpressure e replay. Repo: event_hub_consumer.py — na banca, Kafka no Docker ilustra o padrão.
  • Batch — Data Factory (Glue / Step Functions) via Terraform: janelas de ingestão, landing em Raw (ADLS / S3), validação na borda antes do lake.
  • Batch de perfis (demo): batch_dataprep_mongo.py agrega histórico e grava Cosmos/Mongo — a API consulta na análise online (slide dedicado).
  • Arquitetura Lambda (implementada no desenho): camada speed (streaming → API) + camada batch (lake Medallion + perfis Mongo); serving unificado na API e no painel.

✅ Etapa 5 · Segurança de dados

Defesa em profundidade: Azure (AWS entre parênteses)

  • Criptografia (nativa cloud): TLS; repouso com CMK no Key Vault (KMS + Secrets Manager / SSM).
  • Azure Key Vault (Secrets Manager / SSM + KMS): secrets de Data Factory / Glue, strings de Event Hubs / Kinesis, credenciais Synapse/Cosmos DB (Redshift / DynamoDB) — rotação e identidade gerenciada (IAM roles).
  • Microsoft Entra ID (IAM + IAM Identity Center) + RBAC: papéis por workload; escopos no lake (ABAC S3 / Lake Formation); auditoria (CloudTrail).
  • Mascaramento de dados: aplicado em APIs, pipelines de exportação e ambientes não produtivos — detalhado no próximo bloco.
Demo ao vivo · segurança (camada aplicação)

✅ Etapa 6 · Mascaramento de dados

DataMasker e conformidade LGPD

  • Classe DataMasker completa em src/utils/data_masker.py — reutilizável na API, em jobs e em testes.
  • Campos cobertos: CPF, e-mail, telefone, nome, cartão — padrões configuráveis (últimos dígitos visíveis quando permitido).
  • Anonimização via hash (salt + algoritmo forte) para ambientes analíticos onde o vínculo deve ser irreversível.
  • LGPD: minimização, finalidade, rastreio de acesso; endpoint POST /api/v1/lgpd/mask para demonstração em painel.

✅ Etapa 7 · Arquitetura de dados

Modelo analítico e features

  • Data lake (Medallion): Raw → Processed → Curated com contratos de qualidade; Delta/ACID em Databricks (EMR / Glue + Delta no S3).
  • Data warehouse — Kimball (star schema): fato de transações, dimensões de cliente/canal/tempo; sumarizações em Synapse (Redshift) ou Athena.
  • Features / perfis para serving online: Cosmos DB (MongoDB local) com documento user_profiles por user_id; Gold no lake alimenta retreino; perfis alimentam /analyze.
  • Processamento distribuído: Databricks (EMR / Glue Spark) no ADLS Gen2 (S3).

✅ Etapa 8 · Escalabilidade

Pronto para crescimento

  • Auto-scaling: TU/partições Event Hubs (Kinesis shards / MSK), workers Databricks (EMR), RU/s Cosmos DB (DynamoDB WCU/RCU), pools Synapse (Redshift).
  • Particionamento: lake ADLS/S3; shuffle no Spark; partition key em Cosmos DB (DynamoDB) sem hotspot.
  • Escalonamento horizontal: API em AKS / Container Apps (EKS / ECS / Fargate); novos consumidores / consumer groups.
  • Capacidade-alvo: 10M+ transações/dia com margem para picos (Black Friday, PIX em massa) — validação por teste de carga e custo (FinOps).
